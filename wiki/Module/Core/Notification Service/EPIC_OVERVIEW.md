# Epic Overview: Notification Service

## 1. Business Context

The Notification Service provides the **platform-level notification primitive** for Arobid. Any product module — Chat, Orders, TradeXpo, B2B Marketplace, or future modules — can deliver in-app notifications to users by integrating with this service, without re-implementing notification storage, delivery, or the Notification Center UI.

The service is responsible for:
- Receiving notification events from any module
- Storing notifications per recipient
- Delivering notifications to users in real time
- Surfacing all notifications in a unified **Notification Center** accessible from the global navigation

This service has **no knowledge of module-specific business rules**. It does not know what a "chat message" or a "payment confirmation" means. Modules define their own notification titles, body text, and navigation destinations — the service only handles storage, delivery, and read state.

---

## 2. Scope

| In Scope | Out of Scope |
|---|---|
| Receiving and storing notification events from any module | Email / SMS / mobile push notifications |
| Notification Bell icon with real-time unread count badge | User notification preference / mute settings |
| Centralized Notification Center panel | Admin broadcast or announcement notifications |
| Mark individual notification as read | Notification expiry or auto-dismissal |
| Mark all notifications as read | Module-level notification filtering |

---

## 3. Integration Contract

Any module that wants to deliver a notification to a user must provide the following payload. The Notification Service stores it and delivers it to the recipient — the service does not interpret or transform the content.

### Notification Event Payload (provided by the calling module)

| Field | Type | Required | Description |
|---|---|---|---|
| `userId` | String | Yes | ID of the recipient user |
| `source` | String | Yes | Identifier of the source module. Free-form string defined by each module (e.g., `"chat"`, `"orders"`, `"tradexpo"`). Used by the Notification Center to render a source icon. |
| `type` | String | Yes | Event type within the source module. Free-form string defined by each module (e.g., `"message_received"`, `"payment_confirmed"`). Not interpreted by the service. |
| `title` | String | Yes | Short notification title shown in the Notification Center (max 80 characters) |
| `body` | String | Yes | Preview text shown below the title (max 120 characters) |
| `deepLinkPath` | String | Yes | In-app navigation path to the relevant context (e.g., `/chat/conversations/abc123`). The Notification Center uses this path when the user clicks the notification. |
| `referenceId` | String | No | Optional ID of the related entity. Used for deduplication — see Key Product Rules. |
| `referenceType` | String | No | Optional entity type label (e.g., `"Conversation"`, `"Order"`). Not interpreted by the service. |

### Stored Notification

The Notification Service generates and appends the following fields before persisting:

| Field | Type | Description |
|---|---|---|
| `notificationId` | String | Unique identifier generated by the service |
| `isRead` | Boolean | Read state — `false` by default |
| `createdAt` | DateTime | Timestamp when the notification was created |
| `readAt` | DateTime, nullable | Timestamp when the notification was marked read; `null` if unread |

> The full stored record is the union of the event payload fields and the service-generated fields above.

## 5. Dependencies

| Dependency | Direction | Note |
|---|---|---|
| Authentication / Identity | Upstream | `userId` on the notification event must resolve to a valid authenticated user |
| Any platform module | Upstream (integrator) | Modules emit notification events by providing the Integration Contract payload; the service has no direct knowledge of the emitting module |

---

## 6. Key Product Rules

| Topic | Rule |
|---|---|
| Module agnosticism | The Notification Service does not interpret `source`, `type`, `referenceType`, or `deepLinkPath`. These are opaque strings provided by the calling module and passed through as-is. |
| Recipient scoping | Notifications are per-user and private — a user can only see their own notifications |
| Persistence | Notifications are stored and remain visible in the panel until explicitly marked as read; no auto-expiry in MVP |
| Read state | A notification can transition from unread → read only; the read state cannot be reversed |
| Deduplication | If a module provides both `referenceId` and `referenceType`, the service will not create a duplicate notification for the same `(userId, source, type, referenceId)` combination within a configurable deduplication window |
| Real-time delivery | New notifications are delivered to the recipient's Notification Center without requiring a page refresh |
| Navigation validity | The service stores `deepLinkPath` as provided. Whether the destination still exists at click time is the responsibility of the receiving page, not the Notification Service. |
